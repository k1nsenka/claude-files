# テクニカルライティングスキル

## 概要

このスキルは、技術文書・ビジネス文書を「構造化された思考」として執筆するためのフレームワークを提供する。核となるのは「黄金フォーマット」と呼ばれる4領域モデルであり、**領域の内容よりも領域間の関係性**を重視する点が特徴である。

## クイックリファレンス

| タスク | アプローチ |
|--------|------------|
| メモ・アイデアの構造化 | 4領域に分類し、関係性を検証する |
| 新規文書の作成 | テンプレートを選択し、4領域を順に埋める |
| 既存文書のレビュー | 関係性チェックリストで検証する |
| 文章のリファイン | 一文一義・主述近接・具体化を適用する |

## 黄金フォーマット：4領域モデル

### 4つの領域の定義

| 領域 | 名称 | 記述内容 | 問いかけ |
|------|------|----------|----------|
| **領域1** | 背景・前提 | 従来の状況、既存の状態、前提条件 | 「これまで何が起きていたか？」 |
| **領域2** | 課題・仮説 | 直面する問題、それが課題である理由、原因の仮説 | 「何が問題で、なぜ問題なのか？」 |
| **領域3** | 手段・行動 | 解決アプローチ、その選択理由、意義 | 「何をどうやって解決するか？」 |
| **領域4** | 効果・結論 | 得られる結果、その根拠、もたらす変化 | 「結果として何が言えるか？」 |

### 領域間の関係性（最重要）

**「ボックスの内容よりも、ボックス同士の関係が大切」** — 藤田肇

| 関係 | 領域間 | 説明 | 接続テスト |
|------|--------|------|------------|
| **逆説関係** | 1 ↔ 2 | 背景と課題が相反する | 「しかし」「ところが」で繋がるか |
| **差分関係** | 1 → 3 | 手段は背景との差分のみを記載 | 領域1に存在しない新要素か |
| **順接関係** | 3 → 4 | 手段から効果への因果関係 | 「したがって」「その結果」で繋がるか |
| **裏返し関係** | 2 ↔ 4 | 課題と結論が表裏一体 | 課題が解決された状態になっているか |

## 関係性検証の手順

### Step 1: 逆説関係の検証（領域1 ↔ 領域2）

```
検証方法: 領域1と領域2を「しかし」で接続して読み上げる
```

**合格例:**
- 領域1: 「自転車は移動手段として広く普及している」
- 領域2: 「**しかし**、自転車走行中はラジオを聴くことができない」
- → 普及（正）と利用不可（負）が相反している ✓

**不合格例:**
- 領域1: 「AIの研究が進んでいる」
- 領域2: 「**しかし**、AIの研究をさらに進めたい」
- → 相反関係がない。単なる継続の希望 ✗

### Step 2: 差分関係の検証（領域1 → 領域3）

```
検証方法: 領域3の要素が領域1に存在しないことを確認する
```

**合格例:**
- 領域1: 「自転車が普及」「ラジオが娯楽として利用」
- 領域3: 「ハンドルにラジオを搭載する」
- → 「搭載」は領域1に存在しない新要素 ✓

**不合格例:**
- 領域1: 「既存システムでデータ分析を実施」
- 領域3: 「データ分析を実施する」
- → 領域1と同じ内容。差分がない ✗

### Step 3: 順接関係の検証（領域3 → 領域4）

```
検証方法: 領域3と領域4を「したがって」で接続して読み上げる
```

**合格例:**
- 領域3: 「知識蒸留でモデルを軽量化する」
- 領域4: 「**したがって**、推論速度が5倍向上した」
- → 手段と効果に因果関係がある ✓

**不合格例:**
- 領域3: 「新しいアルゴリズムを導入した」
- 領域4: 「**したがって**、ユーザー満足度が向上した」
- → 因果の飛躍がある。中間説明が必要 ✗

### Step 4: 裏返し関係の検証（領域2 ↔ 領域4）

```
検証方法: 領域4が領域2の課題を解決した状態かを確認する
```

**合格例:**
- 領域2: 「リアルタイム処理が困難」
- 領域4: 「リアルタイム処理が可能になった」
- → 課題の裏返し ✓

**不合格例:**
- 領域2: 「コストが高い」
- 領域4: 「品質が向上した」
- → 別の話題になっている ✗

## よくある失敗パターン

### パターン1: エッセイ型（関係性の欠如）

| 領域 | 失敗例 | 問題点 |
|------|--------|--------|
| 1 | 「××についてやりました」 | 具体性がない |
| 2 | 「△△が気になりました」 | 領域1と相反していない |
| 3 | 「□□を試してみました」 | 選択理由がない、差分が不明 |
| 4 | 「◇◇という結果でした」 | 領域2・3との対応が不明 |

### パターン2: 読書感想文の罠

| 避けるべき | 推奨 |
|------------|------|
| 思ったままに書く | 構造化して書く |
| 時系列で書く | 論理構造で書く |
| 感想を書く | 分析を書く |
| 自由な書式 | フォーマットに従う |

## 文章作成の基本ルール

### ルール1: 一文一義

一つの文には一つの主張のみを含める。

**修正前:**
> 今回の実験では精度が向上したが、計算コストも増加したため、実用性の観点からは検討が必要である。

**修正後:**
> 今回の実験では精度が向上した。一方で、計算コストも増加した。実用性の観点からはトレードオフの検討が必要である。

### ルール2: 主語と述語を近づける

主語と述語の間に過度な修飾を挟まない。

**修正前:**
> モデルAは、事前学習にWikipediaとBookCorpusを使用し、ファインチューニングにはタスク固有のデータを用いて、最終的に85%の精度を達成した。

**修正後:**
> モデルAは最終的に85%の精度を達成した。事前学習にはWikipediaとBookCorpusを使用した。ファインチューニングにはタスク固有のデータを用いた。

### ルール3: 具体的な数値を使う

| 避ける表現 | 推奨表現 |
|------------|----------|
| 大幅に向上した | 15%向上した（78% → 93%） |
| 多くのユーザー | 1,200名のユーザー |
| 短時間で | 3分以内で |

### ルール4: 接続詞の適切な使用

| 関係 | 接続詞 | 使用場面 |
|------|--------|----------|
| 逆説 | しかし、ところが、一方で | 領域1 → 領域2 |
| 順接 | したがって、その結果、ゆえに | 領域3 → 領域4 |
| 並列 | また、さらに、加えて | 同一領域内での列挙 |
| 例示 | 例えば、具体的には | 根拠の説明 |

## 文書タイプ別テンプレート

### テンプレート1: 研究メモ

```markdown
# [研究テーマ]

## 要点（3行サマリー）
1. [課題]: ...
2. [手段]: ...
3. [効果]: ...

## 領域1：背景・前提
[従来の状況、既存手法、前提知識を記述]

## 領域2：課題・仮説
[「しかし」で始められる問題と、その原因仮説を記述]

## 領域3：手段・行動
[領域1との差分となる新要素と、選択理由を記述]

## 領域4：効果・結論
[「したがって」で始められる結果と、領域2の課題が解決された状態を記述]

## 参考文献
```

### テンプレート2: 障害報告書

```markdown
# 障害報告：[概要]

## 領域1：背景・前提
- 対象システム: 
- 通常時の状態: 
- 前提となる構成: 

## 領域2：課題・仮説
- 発生した問題: 
- 影響範囲: 
- 原因の仮説: 

## 領域3：手段・行動
- 実施した対応: 
- その対応を選んだ理由: 

## 領域4：効果・結論
- 対応結果: 
- 復旧確認方法: 
- 再発防止策: 

## タイムライン
| 時刻 | 事象 |
|------|------|
| HH:MM | 障害発生 |
| HH:MM | 復旧確認 |
```

### テンプレート3: 提案書

```markdown
# [提案タイトル]

## エグゼクティブサマリー
[3文で要約：課題 → 手段 → 効果]

## 領域1：現状
[現在の状況、経緯、前提条件]

## 領域2：課題
[解決すべき問題、定量的な影響、原因仮説]

## 領域3：提案内容
[解決策の概要、選択理由、実施計画]

## 領域4：期待効果
[定量的な効果、効果が得られる理由、組織への影響]

## 補足情報
- コスト見積もり
- リスクと対策
- スケジュール
```

## 検証チェックリスト

### 各領域の完成度

- [ ] 領域1: 前提条件が明確に記述されているか
- [ ] 領域1: 従来の状況が具体的か
- [ ] 領域2: 課題が明確に定義されているか
- [ ] 領域2: なぜそれが課題なのか理由があるか
- [ ] 領域2: 仮説が立てられているか
- [ ] 領域3: 解決手段が具体的か
- [ ] 領域3: その手段を選んだ理由があるか
- [ ] 領域4: 結果から言えることが明確か
- [ ] 領域4: なぜそう言えるか根拠があるか

### 領域間関係の検証

- [ ] 領域1 ↔ 領域2: 「しかし」で繋がる逆説関係か
- [ ] 領域1 → 領域3: 差分のみを記載しているか
- [ ] 領域3 → 領域4: 「したがって」で繋がる順接関係か
- [ ] 領域2 ↔ 領域4: 課題と結論が裏返しの関係か

## 補助フレームワーク: PREP法

短い説明や口頭発表では、PREP法を使用する。

| 要素 | 内容 |
|------|------|
| **P**oint | 結論（最も伝えたいこと） |
| **R**eason | 理由（なぜそう言えるのか） |
| **E**xample | 具体例（根拠となる事実・データ） |
| **P**oint | 結論（再提示） |

**使い分け:**
- 黄金フォーマット → 論文、レポート、提案書など構造化文書
- PREP法 → プレゼン、口頭説明、短いメモ
