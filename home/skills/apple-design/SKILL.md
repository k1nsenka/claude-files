# Apple Design

<!-- 出典: https://github.com/emilkowalski/skills/blob/main/skills/apple-design/SKILL.md の日本語訳 -->

Apple がどのようにして「コンピュータを操作している感覚」を消し、「自分自身の延長」と感じられる
インターフェイスを作るか。この知見は Apple の WWDC デザイントーク — 主に *Designing Fluid
Interfaces*（WWDC 2018）— を蒸留し、Web プラットフォーム（CSS、Pointer Events、
`requestAnimationFrame`、Motion/Framer Motion などのスプリングライブラリ）向けに翻案したもの。

貫く原則: **モーションが現在の画面上の値から始まり、ユーザーの速度を受け継ぎ、慣性を前方へ投影し、
どの瞬間でも掴んで反転できる時、インターフェイスは生きていると感じられる。** スプリングは本質的に
割り込み可能で速度を扱えるため、これらすべてを自然に実現する道具になる。

## 核心となる考え方

> 「インターフェイスを人間の思考と動きの仕方に揃えると、魔法のようなことが起きる —
> コンピュータを操作している感覚が消え、自分自身のシームレスな延長のように感じられ始める。」

インターフェイスが流体的（fluid）であるとは、物理世界のように振る舞うこと。つまり、即座に応答し、
連続的に動き、慣性を持ち、境界で抵抗し、動きの途中で向きを変えられる。以下はすべて、そこへ
近づくための手段である。

Apple はデザインを 4 つの人間的ニーズに奉仕するものと捉える:
**安全/予測可能性、理解、達成、喜び。** ここにあるすべてのルールはこのいずれかに奉仕する。

## 1. Response — レイテンシを殺す

ラグが現れた瞬間、直接操作している感覚は「崖から落ちるように」失われる。Response は
他のすべての土台である。

- **pointer-down で応答する。release まで待たない。** ボタンは押された瞬間にハイライトする。
  `click`/touch-up までフィードバックを遅らせると死んだ感触になる。
- **あらゆるレイテンシに警戒する。** debounce、人工的なタイマー、トランジション待ち、
  約 300ms のタップ遅延を監査する。入力経路上にある必須でないものはすべて劣化要因。
- **フィードバックは操作の*最中*に連続して返す。終了時だけではない。** ドラッグ・スライダー・
  ドロワーでは、最初から最後までポインタと 1:1 で UI を更新し続ける — ジェスチャ完了時に
  だけアニメーションさせてはならない。

```css
/* フィードバックは押下に宿る。しかも即座に */
.button:active {
  transform: scale(0.97);
  transition: transform 100ms ease-out;
}
```

## 2. 直接操作 — 1:1 トラッキング

> 「タッチとコンテンツは一緒に動くべきだ。」

ユーザーが何かをドラッグしたら、要素は指に貼り付いたままでなければならない — しかも
*掴んだ位置*からのオフセットを尊重する。掴んだ瞬間に要素の中心へスナップさせると、
錯覚は即座に壊れる。

- Pointer Events と `setPointerCapture` を使い、ポインタが要素の境界の外へ出ても
  トラッキングを継続する。
- 現在の座標だけでなく、直近数回の `pointermove` の**速度/位置の履歴**を記録する —
  リリース時に速度が必要になる。

```js
el.addEventListener('pointerdown', (e) => {
  el.setPointerCapture(e.pointerId);
  const grabOffset = e.clientY - el.getBoundingClientRect().top; // 掴んだ位置を尊重する
  // ...速度算出のため位置 + タイムスタンプの履歴を記録する
});
```

## 3. 割り込み可能性 — 最も重要なただ一つの原則

> 「思考とジェスチャは並行して起きる。」

すべてのアニメーションは、どの瞬間でも割り込み・方向転換できなければならない。ユーザーは
動いている要素を飛行中に掴み、アニメーションの終了を待たずに反転できる必要がある。
閉じかけのモーダルを掴み直したら指に追従するべきで、閉じ終わってから開き直すのではない。

- **トランジション中に入力をロックしない。**
- **常に *presentation*（現在）の値からアニメーションを開始する。目標値からではない。**
  割り込み時は要素の画面上のライブな transform を読み取り、そこから新しいアニメーションを
  開始する。論理値/目標値から始めると目に見えるジャンプが起きる。
- **ジェスチャ駆動のものに CSS transition や `@keyframes` を使わない** — 飛行中に滑らかに
  掴んで反転できないため。スプリングはデフォルトで現在値からアニメーションするので、
  割り込みに必要な性質をそのまま備えている。
- **ジェスチャが反転する時は速度をブレンドする。ハードカットしない。** 反転時にアニメーションを
  別のものへ差し替えると速度の不連続 —「レンガの壁」— が生じる。再ターゲット時に速度を
  引き継ぐスプリングライブラリはこれを回避する。（iOS の *additive animations* がネイティブに
  行っていること。Web では現在の速度から再ターゲットするスプリングライブラリを選ぶ。）
- **2D の動きは独立した X・Y のスプリングに分解する。** 2D 距離に対する単一のスプリングは、
  X と Y の速度が異なる時に同期を失う。

## 4. アニメーションより挙動 — スプリングを使う

> 「アニメーションを、インターフェイスが規定するものではなく、あなたとオブジェクトの間の
> 会話として考える。」

事前に台本化された固定時間のアニメーションは新しい入力に応答できない。スプリングは応答できる —
新しい入力は単に目標を変えるだけで、動きは連続のまま保たれる。ユーザーが触れられるものには
スプリングを使う。

Apple は物理の三つ組（mass/stiffness/damping）を、意図的に 2 つのデザイナー向けパラメータへ
置き換えた。この 2 つで考える:

- **Damping ratio** — オーバーシュートの制御。`1.0` = 臨界減衰。バウンスせず滑らかに整定する。
  `< 1.0` = オーバーシュートして振動する。低いほど弾む。
- **Response** — 値が目標へ到達する速さ（秒）。低いほど機敏。**これは「duration」ではない** —
  スプリングに固定の持続時間はなく、整定時間はパラメータから創発する。

**デフォルト:**
- ほとんどの UI は **damping `1.0`**（臨界減衰）から始める — 優雅で気が散らない。
- バウンス（**damping 約 `0.8`**）を加えるのは、**ジェスチャ自体が慣性を持っていた時だけ**
  （フリック、投げ、ドラッグのリリース）。ただフェードインしただけのメニューのオーバーシュートは
  違和感があり、フリックしたカードのオーバーシュートは正しく感じられる。

**Apple が実際に出荷している値:**

| 操作 | Damping | Response |
| --- | --- | --- |
| 移動 / 再配置（例: PiP） | `1.0` | `0.4` |
| 回転 | `0.8` | `0.4` |
| ドロワー / シート | `0.8` | `0.3` |

**Web への対応（Motion / Framer Motion）:** `bounce` + `duration` のスプリング API は Apple の
damping + response に近い対応を持つ。安全なハウススタイルは、デフォルトで全面的に
`damping: 1.0` 相当のスプリングを使い、bounce は慣性駆動の物理的な操作にだけ確保すること。

```js
import { animate } from 'motion';

// 臨界減衰のデフォルト（オーバーシュートなし）
animate(el, { y: 0 }, { type: 'spring', bounce: 0, duration: 0.4 });

// 慣性のある操作 — 少しだけバウンス。直前にフリックがあったからこそ
animate(el, { y: target }, { type: 'spring', bounce: 0.2, duration: 0.4 });
```

## 5. 速度の受け渡し — ドラッグとアニメーションの継ぎ目

ジェスチャが終わった時、アニメーションは**指の正確な速度のまま継続**しなければならない。
ドラッグとアニメーションの間に見える継ぎ目を作らないためである。この細部こそが「fluid」と
「悪くない」を最も分ける。

ポインタのリリース速度をスプリングの初速として渡す。スプリング API によっては**相対**速度を
要求する — その場合は目標までの残距離で正規化する:

```
relativeVelocity = gestureVelocity / (targetValue − currentValue)
```

例: 要素が `y=50`、目標が `y=150`（残り 100px）、指の速度が 50px/s → スプリングの初速 =
`50 / 100 = 0.5`。Framer Motion / Motion は絶対値の px/s 速度を直接受け取る（`velocity`
オプション）ため、通常は生の値をそのまま渡す。

## 6. 慣性の投影 — ジェスチャが*向かう先*へアニメーションする

> 「小さな入力から大きな出力を作る。」

*リリース地点*から最寄りの境界へスナップしてはならない。速度を使って**静止位置を投影**し —
スクロールの減速とまったく同じ — その投影点に最も近いターゲットへスナップする。これが
フリックに「要素を投げた」感触を与える。

Apple の投影関数そのもの（*Designing Fluid Interfaces* のサンプルコードより）:

```js
// decelerationRate ≈ 0.998 で通常のスクロール感。0.99 でより機敏に
function project(initialVelocity /* px/s */, decelerationRate = 0.998) {
  return (initialVelocity / 1000) * decelerationRate / (1 - decelerationRate);
}

const projectedEndpoint = currentPosition + project(releaseVelocity);
const target = nearestSnapPoint(projectedEndpoint);   // ターゲットは投影点から選ぶ
animateSpringTo(target, { velocity: releaseVelocity }); // その後に速度を受け渡す（§5）
```

注意: 物理の教科書にある `v²/(2·decel)` は Apple が出荷しているものでは*ない* — 上の
指数減衰形を使う。これは良質なボトムシートやカルーセル（Vaul、Embla）の標準的な挙動である。

## 7. 空間的一貫性 — 対称な経路、アンカーされた原点

> 「何かがある方向へ消えたなら、来た場所から現れると私たちは期待する。」

- **入場と退場は同じ経路で。** 右からスライドインしたパネルは右へ消えなければならない。
  右から入って下へ出るのは、つながりが切れて混乱を生む。
- **操作は発生源にアンカーする。** メニュー・ポップオーバー・シートは、それを呼び出した要素から
  発生するべき — `transform-origin` をトリガーに設定し、ボタンとコンテンツの空間的関係を
  明白にする。（ポップオーバーは中心からではなくトリガーからスケールする、という同じ
  原点意識の話。）
- **可逆なトランジションではイージングをミラーする。** 行きの経路と帰りの経路が一致するように
  する（両方向で逆の cubic-bézier 制御点を使う）。

## 8. ジェスチャの方向をほのめかす

人間は軌跡から最終状態を予測する。中間のモーションは行き先を予告するべきだ — Control Center の
モジュールは「指に向かって上へ外へと成長する」。中間フレームは結果を指し示すものにする。
盲目的に補間するだけにしない。

## 9. ラバーバンド — 柔らかい境界

端では、硬く止めるのではなく漸進的に抵抗する。ハードストップは「フリーズした」と読まれ、
連続的な抵抗は「応答しているが、この先には何もない」と読まれる。境界を越えてドラッグするほど
強くなる減衰をかける。

```js
// 境界を越えるほど要素は追従しなくなる — 現実の物は止まる前に減速する
function rubberband(overshoot, dimension, constant = 0.55) {
  return (overshoot * dimension * constant) / (dimension + constant * Math.abs(overshoot));
}
```

## 10. ジェスチャ設計の細部（「感触」チェックリスト）

- **タップ:** touch-*down* でハイライト（即座に）、touch-*up* でコミット。ターゲットの周囲に
  約 10px のヒステリシス/ヒットパディングを設け、ドラッグして離れることでのキャンセルと
  戻ってきての復帰を許す。
- **ドラッグ/スワイプ:** 方向を確定する前に小さな移動しきい値（ヒステリシス、約 10px）を
  要求し、その後は 1:1 で追従する。
- **最初の動きからあり得るすべてのジェスチャを並行して検出**し、意図が明確になったら
  負けた側のジェスチャ認識を迷わずキャンセルする。*最終*状態しか報告しない認識器（`swipeleft` 型のイベント）は
  避ける — フィードバックに必要な連続トラッキングを捨ててしまうため。
- **曖昧性解消の遅延を最小化する。** ダブルタップ検出はシングルタップを不可避に遅らせる。
  ダブルタップが本当に存在する場所でだけそのコストを払う。

## 11. フレームレベルの滑らかさ

滑らかさは*フレームの中身*の問題であり、フレームレートだけの問題ではない。

- フレームごとの位置変化を知覚しきい値以下に保ち、ストロービングを避ける。
- 非常に速い動きには、微かな**モーションブラー / ストレッチ**が速度を符号化し、硬くシャープな
  残像よりも良く読める。
- `requestAnimationFrame` は Web のディスプレイ同期クロック（Apple は `CADisplayLink` を使う）。
  コンポジタに優しいプロパティ — `transform` と `opacity` — だけをアニメーションし、動きが
  差し迫った所には `will-change` でヒントを出す。

## 12. マテリアルと奥行き — 半透明が階層を伝える

Apple は半透明マテリアルを、焦点を奪わずに構造をもたらす浮遊する機能レイヤーとして使う。
Web では `backdrop-filter` で近似する。

- **ナビ / ツールバー / シートは半透明レイヤーとして作る**（`backdrop-filter: blur()` +
  半透明の背景）。コンテンツはその下をスクロールする — 固定の帯を消費する不透明バーにしない。
- **マテリアルの重さが階層を符号化する:** 暗く重いマテリアルは構造領域（サイドバー）を分離し、
  軽いマテリアルはインタラクティブ要素（ボタン）へ注意を引く。**明るい半透明面を別の半透明面の
  上に重ねない** — 可読性が崩壊する。
- **大きな面ほど厚く読ませる:** 小さなチップより強いブラー + 深い影。文脈対応の影も検討する —
  ごちゃついた背景やテキストの上では分離のため重く、無地の背景の上では軽く。
- **集中させるなら dim、フローを保つなら分離。** モーダルなタスクでは面に暗転スクリムを
  組み合わせ、背景を奥/下へ押し込む。並行的で非ブロッキングなパネルではスクリム*なし*で
  半透明とオフセットを使い、フローを壊さない。シートを重ねる場合は親レイヤーを段階的に
  暗くし、奥へ押し込む。
- **Vibrancy が変化する背景上のテキストの可読性を保つ。** ブラー/半透明面の上で平坦なグレーの
  テキストを使わない — より高いコントラスト、やや重いウェイト、わずかな letter-spacing の
  増加を使う。色は半透明の前景ではなくソリッドなレイヤーに置く。
- **ハードな区切り線ではなく scroll edge effect。** スティッキーヘッダーの下の 1px ボーダーの
  代わりに、コンテンツが浮遊クロームと接する場所へ小さなブラー/グラデーションマスクを
  フェードさせる — 浮遊 UI が実際にコンテンツと重なる場所だけに。
- **単にフェードさせず、マテリアライズさせる。** ガラス/ブラーの面では、enter/exit で
  ブラー半径とスケールを一緒にアニメーションし、単なる opacity フェードではなく実在の
  マテリアルが到着したように読ませる。

```css
.toolbar {
  background: rgba(255, 255, 255, 0.6);
  backdrop-filter: blur(20px) saturate(180%);
  border-top: 1px solid rgba(255, 255, 255, 0.4); /* 明るい上端 = マテリアルに当たる光 */
}
```

## 13. マルチモーダルフィードバック — モーション + 音 + ハプティクス

感覚を組み合わせる 3 つのルール（*Designing Audio-Haptic Experiences* より）:

1. **因果性** — 何がそのフィードバックを引き起こしたかが自明であること。実際の因果イベント
   （トグルが切り替わる、アイテムが定位置にスナップする）でトリガーし、フィードバックの
   性格をアクションの物理性に一致させる。
2. **調和** — 視覚・音・ハプティクスは**同じフレーム**で発火する。相互のレイテンシは錯覚を
   破壊する。CSS transition が音やハプティクス（Vibration API）より遅れないようにする。
3. **有用性** — フィードバックは価値のある場所にだけ加える。ハプティクス/音は意味のある瞬間
   （成功、エラー、コミット、スナップ）のために確保する。過剰なフィードバックは、ユーザーに
   すべてを無視することを学習させる。

## 14. Reduced motion とアクセシビリティ

Reduced motion はフィードバックを*なくす*ことではない — より穏やかで、前庭系を刺激しない
等価物を意味する。3 つの独立したシグナルに応答し、コンポーネントに焼き込む:

- **`prefers-reduced-motion: reduce`** — スライド/スプリング/パララックスを短い opacity の
  **クロスフェードや静的トランジション**へ置き換える。弾性/オーバーシュートは落とす。
  理解を助ける opacity や色の変化は残す。
- **`prefers-reduced-transparency: reduce`** — 半透明面をより曇らせる/ソリッドにする:
  背景の不透明度を上げ、ブラーを落とす。
- **`prefers-contrast: more`** — ほぼ不透明の背景 + 明確でコントラストのあるボーダー。

さらに: ビューポート全体で動く背景、遅いループ振動（0.2 Hz 付近 / 5 秒に 1 周期）、急な輝度の
ジャンプ（ダーク↔ライトのテーマ変更はイーズさせる）を避ける。大きく動くオブジェクトは移動中に
半透明にし、大きな再配置の間は大きな面をフェードアウトして、整定後にフェードインし直す。

```css
@media (prefers-reduced-motion: reduce) {
  .sheet { transition: opacity 200ms ease; transform: none !important; }
}
@media (prefers-reduced-transparency: reduce) {
  .toolbar { background: white; backdrop-filter: none; }
}
```

## 15. タイポグラフィ — optical sizing・tracking・leading

Apple はサイズに応じて形が変わるように書体を設計する。同じ規律は Web にも適用される。
（*The Details of UI Typography*、WWDC 2020 より。）

- **Tracking（letter-spacing）はサイズ固有 — 全サイズに 1 つの値は決して使わない。** 大きな
  ディスプレイテキストは*負*の tracking を求める（大きくなるほど字間が開きすぎて見える）。
  小さなテキストは可読性のためわずかに*正*の tracking を求める。固定の `letter-spacing` は
  どこかで必ず間違っている。見出しは詰め、本文は `0` 付近に置く。
- **Leading（line-height）はサイズと逆方向に追従する。** 大きな見出しはタイトに、本文はゆるく。
  アセンダ/ディセンダの高い文字体系（スクリプト）では増やし、密で情報量の多い UI では詰める。
- **階層は weight + size + leading のセットで作る。** サイズ単体ではない。強調は weight で —
  スペースを取らずに存在感を加えられる。
- **ユーザーのテキストサイズ設定を尊重する**（Dynamic Type）。レイアウトをテキストと*一緒に*
  スケールさせる — 余白は固定 px ではなく `rem`/`em` で — フォントが大きくなっても
  レイアウトが壊れないように。
- **カスタム書体の前に、まずプラットフォームのシステムフォントをデフォルトにする。**
  optical sizing、tracking テーブル、可読性チューニングを既に備えている。理由がある時だけ
  上書きする。

```css
:root { font: 100%/1.5 system-ui, sans-serif; } /* 本文: システムフォント、快適な leading */

.display {
  font-size: clamp(2rem, 5vw, 4rem);
  line-height: 1.05;        /* 大きなテキストにはタイトな leading */
  letter-spacing: -0.02em;  /* 大きくなるほど負の tracking */
  font-optical-sizing: auto;
}
```

## 16. デザインの基盤 — 8 つの原則

上記のモーションとクラフトは、Apple の 8 つのデザイン原則（*Principles of Great Design*、
WWDC 2026）に奉仕する。推論に使う語彙としてこれらを使う:

1. **Purpose（目的）。** 意図を持って作る。何を*作らないか*を決める。すべての機能はユーザーの
   時間・注意・信頼を要求する — その予算は見返りのある場所にだけ使う。
2. **Agency（主体性）。** 人をコントロールの座に保つ: 選択肢を提供し、単一の道を強制しない。
   寛容さで裏打ちする — うっかりミスには簡単な undo を。確認ダイアログは真に破壊的で
   不可逆なアクションにだけ（控えめに使う。乱用すると人はクリックで素通りすることを学習する）。
3. **Responsibility（責任）。** ユーザーの利益のために行動する。プライバシー: 適切な瞬間に、
   必要な分だけ、透明性を持って求める。安全: 誤用と危害を予期する — 特に AI では
   （アレルギー対応のレシピアプリは有害な材料を提案してはならない）。プレビュー・確認・
   免責事項を加える。リスクが価値を上回る機能は削る。
4. **Familiarity（親しみ）。** 人が既に知っていることの上に築く。文字通りすぎず抽象的すぎない
   メタファーを使い（ゴミ箱は削除を意味する）、その物理法則を尊重する。一貫させる: 同じに
   見えるものは同じように振る舞い、同じ場所に住む（macOS で閉じるボタンは常に左上）—
   人が次に何が起きるかを予測できるように。親しまれたパターンを破るのは、より良いと証明
   できる時だけ — その時はテストする。仮定で済ませない。
5. **Flexibility（柔軟性）。** 異なる文脈・デバイス・能力の全範囲に向けて設計する。
   プラットフォームに適応し（iPhone = 素早いタッチ操作。デスクトップ = 精密なポインタ制御での
   深いワークフロー）、状況にも適応する。包摂的に設計する（年齢、言語、熟練度、
   アクセシビリティ）。単一のレイアウトが全員に合わない時は、パーソナライズさせる —
   コントロールの並べ替えや、使わないものの非表示。
6. **Simplicity（シンプルさ）— ミニマリズムではない。** 核心の目的が輝くように不要物を
   剥ぎ取る。すべてを一箇所に埋め込むのはミニマルに見えるがシンプルではない。簡潔に
   （平易な言葉、専門用語なし、少ない手順）、明快に（階層 — 順序・余白・コントラスト — を
   使い、最も重要なものが最も目立つように）。すべての要素は居場所を稼ぐ。時には*足す*ことが
   シンプルにする（残り時間を表示する動画のスクラバー）。一般的な経路を最初に見せ、高度な
   オプションは一段深くに置く。
7. **Craft（クラフト）。** 妥協のない細部への注意が信頼を築く。美しいタイポグラフィ、
   ライト/ダークに適応する色、明快なアイコノグラフィ、即座で自然なフィードバックを返す
   応答的なアニメーション。何一つランダムではない — すべての余白・タイミング・整列の値は、
   擁護できる意図的な選択である。ガタつくスクロール、ずれたアイコン、回転で壊れる
   レイアウトは不注意と読まれる。クラフトには反復と寿命が要る — 機能とハードウェアの変化に
   合わせて設計を進化させ続ける。
8. **Delight（歓喜）。** 他の 7 つを正しくやり遂げた結果であり、上から貼り付けた紙吹雪では
   ない。人に感じてほしい感情（穏やかさ、自信、興奮）を決め、すべての決定でそれを強化する。

これらに奉仕する戦術的ルール:

- **フィードバックには 4 種類ある:** ステータス、完了、警告、エラー。意味のあるアクションを
  確認し、進行中のステータスを露出し、問題が起きる前に警告し、インラインで検証する
  （submit 時ではなく）。
- **Wayfinding（道案内）。** すべての画面はこう答えるべき: 私はどこにいる? どこへ行ける?
  そこには何がある? どうやって出る? ユーザーを決して閉じ込めない。
- **グルーピングとマッピング。** 近接は関係を含意する。コントロールは影響を与える対象の
  近くに置き、変化する対象を鏡写しにするように配置する。コントロールの説明にラベルが
  必要なら、マッピングが弱い。
- **直接的で具体的なラベルは、無難で汎用的なラベルに勝る。** ナビ項目は中身の名前で呼ぶ
  （「Progress」「Library」）。曖昧な傘（「Home」）ではなく。具体性が予測可能性を生む。

## 17. プロセス

- **インタラクティブにプロトタイプする — 動くデモは「100 万枚の静的デザイン」に値する。**
  インターフェイスは、作って触って遊ぶことで発見する。動くプロトタイプは具体的な基準にも
  なり、凡庸な最終実装を防ぐ。
- **インタラクションとビジュアルを一緒に設計する。** 「どこで一方が終わり、もう一方が
  始まるのか、分からないべきだ。」モーションはピクセルの後から加えるレイヤーではない。
- **実際の文脈で実際の人とテストする。** そしてモーションを新鮮な目でレビューする —
  スローモーションやコマ送りで再生し、通常速度では見えないものを捕まえる。
