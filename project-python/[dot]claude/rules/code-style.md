# コーディングスタイルルール

## 基本原則
- PEP 8 に従ったコードを書く
- シンプルで読みやすいコード
- 適切な命名（変数、関数、クラスなど）
- 一つの関数は一つの責務を持つ
- エラーハンドリングを適切に実装
- コメントは必要な箇所にのみ付ける

## コード品質ツール（uv経由で実行）

### リンター
```bash
uv run ruff check .
uv run ruff check --fix .
```

### フォーマッター
```bash
uv run ruff format .
```

### 型チェック
```bash
uv run mypy src/
```

## 設定値管理の原則

### 禁止事項
- コード内にハードコードされた設定値（数値、文字列等）の使用を禁止

### 必須事項
- 全ての設定可能なパラメータは設定ファイルまたは設定オブジェクトから取得
- 必要な設定値が存在しない場合は警告ログを出力し、適切なエラーハンドリング
- 使用される設定値をログ出力して正しく反映されていることを確認可能に
- どの設定がどこで使用されているかを明確にする

### 禁止パターン
- 設定読み込み失敗時にデフォルト値でマージ/補完（設定の問題に気づけなくなる）
- 設定読み込みに失敗した場合はエラーとして明示的に報告

## 実装品質管理

### 設計準拠の原則
- 設計書通りの実装を必ず行う（簡易実装・モック実装は禁止）
- 設計書と異なる実装が必要な場合は事前にユーザに許可を取る
- 「簡易実装」「TODO」「モック」等のコメントを残したまま完了とみなさない

### 完全実装の確認
- LLM API呼び出しが必要な機能は実際にAPI呼び出しを実装
- プロンプト定義、レスポンス処理、エラーハンドリングを含む完全な実装
- テスト時も実際の機能テストを行い、モックは最小限に留める
